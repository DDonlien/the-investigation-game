## 对后续 agent 的工作要求

### 1. 优先基于现有资料，不重复整理旧资产

- 不要重新引入 `docx / xlsx`
- 不要重复做“资料清洗”类工作，除非发现明确缺失
- 所有设计判断先以现有 `md / csv` 为依据

### 2. 修改内容时保持 Obsidian 友好

- 新增笔记优先使用 `[[wikilink]]`
- 如果写说明文档，优先链接现有笔记而不是重复复制内容
- 新建索引页时，尽量作为入口页而不是冗余正文

### 3. 面向实现而不是面向归档

后续 agent 的主要产出应当围绕：

- TS 版本游戏的一阶段需求拆解
- 系统设计
- 数据结构
- 前端交互
- 玩法循环落地

而不是继续围绕“文档转格式”。

## 推荐分工

如果后续要多 agent 协作，建议按下面维度分工：

- 系统设计 agent：规则映射、状态机、回合流程、网络边界
- 数据建模 agent：调查员、卡牌、效果、环境、事件的数据结构
- 客户端 agent：UI、页面流、交互、动画、战斗视图
- 游戏逻辑 agent：回合解析、效果结算、判定优先级
- 联机 agent：P2P 同步、房间、状态一致性

## 当前边界

- 目前仓库主要是**设计资料**
- 还不是完整的 TS 游戏工程
- 如果后续开始实现，建议先明确代码落位方式，再逐步把设计资料映射到代码
